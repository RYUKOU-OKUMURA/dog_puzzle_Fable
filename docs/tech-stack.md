# 技術スタック — しばちゃんのおさんぽパズル(仮)

## 1. 方針

- **ブラウザだけで完結**: インストール不要、タブレットでもすぐ遊べる
- **アイソメトリック(クォータービュー)3D**: フル3Dより実装が軽く、カメラ操作不要で子供に優しい。ジオラマ・箱庭のかわいさが出しやすい
- **サーバーレス**: バックエンド不要。静的ホスティングのみで運用コストゼロ
- **プロトタイプ優先**: まず動くものを作り、遊びながら調整する

## 2. コア技術

| 領域         | 技術                  | 選定理由                                                                                |
| ------------ | --------------------- | --------------------------------------------------------------------------------------- |
| 言語         | TypeScript            | 型があることでパズルのロジック(グリッド・パネル接続判定)のバグを防ぎやすい              |
| 3D描画       | Three.js              | Web 3Dのデファクト。アイソメトリックビューは OrthographicCamera(平行投影)で実現         |
| ビルドツール | Vite                  | セットアップが軽く、開発サーバーが高速。静的ビルドでそのままデプロイ可能                |
| UIレイヤー   | HTML/CSS オーバーレイ | 図鑑・メニュー・ボタンは3D内でなくDOMで実装。日本語表示(ふりがな含む)が簡単で開発が速い |

※ フレームワーク(React等)はプロトタイプ段階では不使用。UIが複雑化したら導入を検討。

## 3. ゲーム実装の構成要素

### 3.1 3Dシーン

- `OrthographicCamera` による固定アイソメトリック視点(カメラ操作なし)
- グリッドベースのタイルマップ(例: 8×8)で街を構成
- 建物・木・道などはローポリの箱庭パーツ
- ライティングは環境光+平行光1灯のシンプル構成(パステル調のトーンマッピング)

### 3.2 3Dアセット

| 種類                   | 調達方法                                                                                                |
| ---------------------- | ------------------------------------------------------------------------------------------------------- |
| 街パーツ(建物・木・道) | プリミティブ(BoxGeometry等)の組み合わせで自作、または Kenney(CC0)等のローポリ素材                       |
| 犬モデル               | フェーズ1はプリミティブ組み合わせのデフォルメ柴犬を自作。将来的にGLTFモデル(ライセンス要確認)へ差し替え |
| フォーマット           | glTF (.glb) を標準とする                                                                                |

### 3.3 パズルロジック

- グリッド上のセルに「道パネル」データ(種類・回転)を保持
- 各パネルは接続方向(上下左右)の情報を持ち、隣接セルとの接続をグラフ探索(BFS)でスタート→ゴール判定
- 3D描画とロジックは分離(ロジックは純粋なTSモジュールとして単体テスト可能に)

### 3.4 アニメーション

- 柴犬の歩行: 完成したルートのセル列に沿ってTweenで移動+上下の弾みで「テクテク感」を表現
- ライブラリ: GSAP または Three.js標準の AnimationMixer(モデル導入後)

### 3.5 ステージ定義(20ステージ量産の土台)

- ステージはテキスト地図(ASCIIマップ)で記述し、パーサで `StageDef` に変換する
  - 例: `木` `家` `ビ` などの1文字(または短い記号)が scenery、`★`=スタート、`◎`=ゴール、`空`=パネルスロット、`│` `─` `┐` 等が固定道
  - stage01.ts の冒頭コメントにある手描き地図が、そのままデータになるイメージ
- おやつ・橋などのギミックも地図記号+ステージごとのメタ情報(解放ギミック、持ち札パネル一覧)で表現
- パーサは純粋TSモジュールとしてユニットテスト(不正な地図はビルド時/テスト時に検出)

### 3.6 データ保存(プロフィール対応)

- `localStorage` に JSON で保存。キーをプロフィールIDで分離(例: `save:v2:<profileId>`)
- プロフィール一覧(なまえ・アイコン)は別キーで管理。最大4プロフィール
- 既存のフェーズ1セーブデータは初回起動時に「プロフィール1」へ移行
- アカウント・通信は一切なし。端末間同期もしない

### 3.7 音(フェーズ3)

- 効果音は **Web Audio API の合成のみ**(音声ファイル・howler.js 等の依存追加なし)。詳細は [plan-phase3.md](./plan-phase3.md) M13
- BGMは引き続き見送り(未承認)

## 5. デプロイ(GitHub Pages + PWA)

- **GitHub Pages** で配信(無料プランのためリポジトリは public。個人情報はコードに含めない)
- GitHub Actions(`.github/workflows/deploy.yml`)で main への push 時に `npm test` → `vite build` → Pages へ自動デプロイ
- Vite の `base: '/dog_puzzle_Fable/'`(絶対パス。相対 `./` は使わない — PWA の SW / manifest scope のため)
- **PWA対応**: `vite-plugin-pwa`(devDependency。Vite 8 対応は 1.3.0+)で manifest + Service Worker を生成
  - theme_color / background_color は空の色 `#bde3ff`
  - アイコンは `public/icons/` の自作肉球 SVG / PNG(design-guide 6.2 パレット内)
  - iPad で「ホーム画面に追加」するとフルスクリーンのアプリとして起動
  - 初回ロード後はオフラインでもプレイ可能(アセットをプリキャッシュ)
