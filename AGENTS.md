# AGENTS.md — AIエージェント向け作業ガイド

このリポジトリで作業するAIエージェントは、まずこのファイルを読むこと。
ここに書かれたルールは、タスク指示と矛盾しない限り**必ず**守る。

## 1. プロジェクトの性格(前提として頭に入れること)

- 「しばちゃんのおさんぽパズル」— 小学生の子供のためのブラウザゲーム。**完全に家庭内利用**で、一般公開・収益化はしない
- ただし GitHub Pages 配信のためリポジトリは public。**個人情報(子供の名前・家庭の情報など)をコード・コメント・コミットメッセージ・ステージデータに絶対に含めない**
- サーバーレス・通信なし・課金なし・外部リンクなし。この方針を崩す実装(外部API呼び出し、CDN読み込み、アナリティクス等)は提案もしない
- 遊ぶのは8〜10歳の子供。UX判断に迷ったら「子供が一人で迷わず遊べるか」を基準にする

## 3. コマンド

```bash
npm run dev        # 開発サーバー (Vite)
npm run build      # 型チェック (tsc --noEmit) + 本番ビルド
npm run test       # ユニットテスト (Vitest)
npm run lint       # ESLint
npm run format     # Prettier
```

## 4. アーキテクチャ

```
src/
  core/   パズルの純粋ロジック(グリッド・パネル接続・経路探索)
  game/   ゲーム進行(状態機械・アニメーション制御・パズル操作)
  scene/  Three.js 描画(レンダラ・街・犬・パネルのメッシュ・入力)
  stage/  ステージ定義と犬種データ
  ui/     DOMオーバーレイ(画面・HUD・図鑑)
  save/   localStorage 保存
tests/    core のユニットテスト
```

**依存方向のルール(厳守)**:

- `core/` は他のどの層にも依存しない。DOM・Three.js も import 禁止。純粋TSのみ
- `stage/` はデータ定義。`core/types` 以外に依存しない
- `scene/` はゲームルールを知らない(「何を描くか」だけ)。判定ロジックを scene に書かない
- 全体の配線・状態遷移は `game/state.ts`(状態機械)と `main.ts` に集約する

## 5. コーディング規約

- TypeScript strict。`any` 禁止。既存コードの命名・コメント密度に合わせる
- コードコメントは日本語。「なぜそうするか」だけを書く(既存コード参照)
- UIに出す文字列は**ひらがな・カタカナ中心**。漢字を使う場合は必ず `<ruby>漢字<rt>かんじ</rt></ruby>` 形式(詳細は design-guide)
- 新しい npm 依存の追加は原則しない。必要と判断したら理由を明記して最小限に(候補は tech-stack.md に挙がっているもののみ)
- フレームワーク(React等)は導入しない

## 6. テストと品質ゲート(Definition of Done)

タスク完了と言う前に、以下を**全部**満たすこと:

1. `npm run build` が通る(型エラーゼロ)
2. `npm run test` が全部通る。**`core/` に追加・変更したロジックには必ずユニットテストを書く**(ステージデータのパーサ、接続判定、収集判定など)
3. `npm run lint` がエラーゼロ
4. 見た目に触れた場合: [docs/design-guide.md](docs/design-guide.md) 末尾のセルフチェックリストを通す
5. `npm run dev` で実際に一連のプレイ(パズル→お散歩→出会い→図鑑)を通しで確認する。ビューポートを縦長(iPad縦持ち 820×1180 相当)にしても破綻しないこと
6. 要件・仕様を変えた場合は docs/ の該当ドキュメントも同じコミットで更新する

## 7. やってはいけないこと

- design-guide にない色・フォント・角丸・影を勝手に足す(デザイン変更は人間の承認事項)
- テクスチャ画像・外部3Dモデル・Webフォントの追加(現方針はプリミティブ+単色マテリアルのみ)
- セーブデータ形式の破壊的変更(変更するときは必ず旧形式からの移行コードを書き、移行テストを付ける)
- 「とりあえず動く」ための場当たり的な層またぎ(core から DOM を触る等)
- 子供が読めない文言・怖い/否定的な文言(「しっぱい!」「ミス!」等)をUIに出すこと。失敗時も優しい表現にする(既存の「あれれ? みちが つながって いないみたい」参照)
