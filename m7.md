## 背景

最後の新ギミック「橋」。道が立体交差できるようになり、経路の見通しが
一気に立たなくなる。難易度への寄与が最も大きいマイルストーン。

## 仕様の確定事項(plan.md に無い部分はここで決める)

- 新パネル種 `bridge`: **上=南北(N-S)、下=東西(E-W)が独立して通れる**。
  N→E のような「橋の上下をまたぐ曲がり」は不可
- 橋は**固定道のみ**。プレイヤーは配置できない(`PLAYER_PANEL_KINDS` に入れない。
  `end` と同じ扱い)。理由: パレットUI・回転の意味(90°回しても接続が同じ)・
  難度制御が複雑になる割に得るものが無い。placeable 化はスコープ外とし、
  変えたくなったら人間に確認する
- 地図トークン: 例 `橋`(向きは1種で足りる。上下どちらが南北かは見た目の問題で、
  接続は対称)。plan.md M1 のトークン表とパーサテストも更新する
- **橋マスにおやつは置けない**(どちらの道で取ったか曖昧になるため)。
  `defineStage` で日本語メッセージ付きパースエラーにする

## 現状の実装(前提知識)— ここが本タスクの核心

- 接続判定・経路探索は `src/core/panel.ts` と `src/core/path.ts`。
  現在の `findPath` は「セル=ノード」の BFS で、隣接セル双方が互いに向けて
  接続を持てば通れる。状態は「(マス, 取ったおやつのビットマスク)」
- **この「セル単位の接続集合」では橋を表現できない**。橋セルは N↔S と E↔W のみ
  通れて N↔E は通れないため、「どの向きから入ったか」で出られる向きが変わる。
  → BFS の状態を「(マス, 進入方向, おやつマスク)」に拡張するのが推奨
  (橋以外のパネルは「持っている接続同士は全部行き来できる」として一般化すれば
  既存挙動と同値)。状態数は高々 マス数×4方向×8マスクで十分小さい
- **route に同じ座標が2回現れうる**(橋の上と下を両方通るルート)。
  route(セル列)を消費する側が「セル一意」を仮定していないか必ず確認する:
  `src/game/walk.ts`(歩行)、`src/scene/treats.ts`(ぱくっ演出)、
  首かしげ演出の部分ルート
- ★◎の向き自動導出(`src/stage/stageMap.ts` の decideDirection 相当)が
  橋トークン隣接で正しく動くかも確認する
- 解可能性テストの見本: `tests/w3stages.test.ts`(M6で作った palette 対応ソルバ)

## 変更内容(この順で。各ステップでテストを書いてから次へ)

1. **core: 橋の接続モデルと経路探索**(`src/core/`、純粋TSのみ)
   - `PanelKind` に `bridge` を追加。接続を「通れる方向ペア」で表せるようにし、
     既存4種は従来と同値、橋は {N↔S, E↔W} とする
   - `findPath` を進入方向つき状態に拡張。ユニットテスト必須。最低限:
     橋なし盤面で従来と完全に同じ結果 / 橋を直進で通過(N→S、E→W それぞれ) /
     **橋で曲がれないこと**(N→E が経路にならない盤面) /
     同一の橋マスを上下2回通るルートが正しく復元される /
     おやつ+橋の複合(橋の先のおやつ)
   - 橋は回転・除去不可(固定なので既存の fixed ガードで足りるはずだが、テストで保証)
2. **stage: 橋トークンのパース**(`src/stage/stageMap.ts`)
   - トークン `橋` → fixed な bridge パネル。橋マスへの treats 指定はパースエラー。
     パーサテスト追加
3. **scene: 橋の3D表現と歩行**
   - 下の道の上をアーチ橋がまたぐ。プリミティブ+単色のみ、**高さ1.5以内**
     (design-guide 7章)。色は 6.2 の色域内
   - しばちゃんは橋の上下どちらも歩ける。上を通るときは高さを滑らかに補間
     (どちらを通るかは route から game/ が決める。**scene に判定ロジックを書かない**)
4. **W4 の4ステージ**(w4-s1〜w4-s4、難度表どおり 🦴3・4・4・5)
   - w4-s1: **橋導入**。橋1、交差1回だけの「ルールを覚える回」(ノコギリ型カーブ)
   - w4-s2: 橋1+おやつ2 / w4-s3: 橋2+ダミー / w4-s4: 橋2+おやつ3
   - **橋を使わないと解けない設計を最低2つ**含めること。解可能性テスト
     `tests/w4stages.test.ts` では「クリア可能」に加え、その2ステージについて
     「橋マスを通らないルートでは complete にならない」ことをアサートする
   - メキシコ・アメリカ風の街パーツを新添景トークンで追加(例: `サ`=サボテン)。
     トークン表・パーサテストも更新
5. **犬データ4種**: チワワ / ボストンテリア / アラスカンマラミュート /
   ラブラドールレトリバー。正確な情報+ふりがな(`<ruby>` 必須)。`scale` は実体格どおり
6. **カタログ登録**: `catalog.ts` の w4 に4ステージを設定
